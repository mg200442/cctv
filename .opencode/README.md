# OpenCode Portable Workspace

This folder is a clean, project-agnostic OpenCode operating system template.
It is intentionally isolated from the application code and stores all OpenCode
agents, commands, skills, gates, task queues, reports, reviews, and planning
templates under `.opencode/`.

## Goal

- Start a brand-new project with the same gated workflow.
- Drop the folder into an existing project without mixing OpenCode files with application docs.
- Keep generated OpenCode artifacts inside `.opencode/`.
- Avoid project-specific product, architecture, or implementation content.

## Activate in a new or existing project

OpenCode natively scans `.opencode/agents`, `.opencode/commands`, and `.opencode/skills`.
Place this folder at the project root as `.opencode/`, then copy the base project config
to the root if the project does not already have an `opencode.json`:

```sh
cp .opencode/opencode.project.json opencode.json
```

Then restart OpenCode from the project root:

```sh
opencode
```

The prompts write workflow artifacts under `.opencode/` so project files stay isolated.

## Daily Use

Start with `.opencode/project-status.md`. It is the single operational dashboard for
current state, blockers, next actions, approvals, active slices, and completed work.
Answer agent questions in `.opencode/open-questions.md`.

On a virgin workspace, run `/bootstrap-workspace` first. To add work later, run
`/add-task`; it will ask the needed questions conversationally and update the dashboard.
For model/provider setup, run `/configure-models`; it recommends per-agent model
assignments and applies them only after approval.
For work that should live on its own Git branch, use `/prepare-workstream-branch` or
let `/add-task` ask for branch preference.
For external designs, add design files to `.opencode/design-handoff/` and run
`/prepare-design-handoff`. It processes the inbox, archives the package, and resets
`context.md` for the next design.
When a design is already prepared, treat it as the build contract after approval:
study every supplied screen, state, asset, prototype, exported HTML/CSS, responsive
variant, and technology note before architecture, slicing, or implementation.

## Folder map

- `agents/`: primary orchestrator and specialist subagents.
- `commands/`: slash-command prompts for gates, slices, reports, and auto-advance.
- `skills/`: reusable task behavior.
- `gates/`: gate templates and exit criteria.
- `project-status.md`: single user-facing dashboard for state, approvals, blockers, next actions, and slices.
- `design-handoff/`: external design package and `context.md` used as the project starting point when design comes from another tool or designer.
- `tasks/`: supporting backlog, roadmap, compatibility files, and slice templates.
- `docs/`: OpenCode-owned product, architecture, security, testing, infra, and decision templates.
- `reports/`: report template and dated reports generated by workflow.
- `reviews/`: review templates and review outputs.
- `model-profiles.md`: guidance for choosing global and per-agent models.

## Virgin state

All non-runtime content is either a template, placeholder, or empty scaffold. Replace bracketed
placeholders such as `[project-name]`, `[owner]`, and `[date]` when starting a real project.

## Design-first projects

If the project starts from external designs, place the handoff in `.opencode/design-handoff/`
and complete `.opencode/design-handoff/context.md` first. Gate 2 and Gate 3 should use
that context as the design source of truth before architecture or implementation begins.
If the handoff names technologies, frameworks, component libraries, assets, exported code,
or design-system constraints, Gate 4 and Gate 5 must map the implementation to those
inputs. Approved designs should be implemented with high visual and behavioral fidelity;
allowed deviations must be documented before implementation, and complex design builds
may be split into as many small vertical slices as needed.

## Model flexibility

`.opencode` is provider-flexible by default. The base config has one global model in
`.opencode/opencode.project.json`, and agents inherit it unless a specific `model:` line is
added to an agent file. See `.opencode/model-profiles.md` for balanced, low-cost, and
high-accuracy strategies.
